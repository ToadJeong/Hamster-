# 햄찌랜드 개발·운영 원칙

이 문서는 햄찌랜드를 만들 때 항상 지켜야 하는 7가지 원칙을 정리합니다.
**모든 신규 기능·PR은 이 원칙을 충족하는지 점검 후 머지합니다.**

---

## 1. 빌드 전략 — 웹앱 우선, Capacitor로 모바일 패키징

- 메인 제품은 **Next.js 웹앱**(`apps/web`)이며, 모바일 앱은 동일한 웹 자산을 **Capacitor**로 감싸 배포한다.
- 별도 React Native 앱(`apps/mobile`)은 보조 옵션이며, 핵심 기능을 먼저 웹에서 검증한 뒤 추가한다.
- 신규 화면 추가 시 점검:
  - 모바일 뷰포트(360px)에서 레이아웃이 깨지지 않는가?
  - 가로 스크롤이 발생하지 않는가?
  - 터치 타깃이 최소 44×44px인가?
- Capacitor 패키징 절차는 [`/docs/capacitor-packaging.md`](./capacitor-packaging.md) 참고.

## 2. 출시 필수 페이지를 “기능 개발 단계”에 미리 만들어둔다

다음 페이지는 기능 출시 전이라도 **반드시 존재**해야 한다.

| 페이지 | 경로 | 비고 |
|---|---|---|
| 개인정보 처리방침 | `/privacy` | 관리자 화면에서 본문 편집 가능 |
| 이용약관 | `/terms` | 동일 |
| 데이터 삭제 요청 | `/account/delete` | 익명 글 삭제 안내 포함 |
| 공지 영역 | 홈 상단 배너 | `site.notice` site_settings로 즉시 갱신 |

신규 페이지가 사용자 정보를 다룬다면, 그 PR에서 **위 페이지의 본문도 갱신**해야 한다.

## 3. iOS 심사·앱 출시 체크리스트

신규 앱/주요 업데이트 출시 전 [`/docs/release-checklist.md`](./release-checklist.md)를 모두 통과해야 한다.

핵심:
- 심사용 테스트 계정(이메일/비밀번호) 준비
- 로그인 흐름을 1줄로 설명할 수 있는가
- 익명 게시 기능이 있는 경우, 익명 우회 경로도 설명에 포함

## 4. 결제·구독은 “결제 후 무엇이 열리는지” 문구를 명시한다

- 현재 햄찌랜드에는 결제 기능이 없다.
- 추후 도입 시 다음을 반드시 노출:
  1. 결제 처리 주체(Apple/Google/자체)
  2. 무엇이 잠금 해제되는가
  3. 환불·복원 방법
- 위 문구는 코드에 박지 않고 `site_settings`의 `legal.billing_html` 같은 키로 빼둔다.

## 5. 푸시 알림은 시나리오를 먼저 정한다

- 알림을 보내기 전, [`/docs/push-notification-plan.md`](./push-notification-plan.md)에 등록된 시나리오에 부합하는지 확인한다.
- 시나리오에 없는 알림은 보내지 않는다.
- 시나리오 추가는 PR로 합의한다.

기본 시나리오 3개:
1. 내가 쓴 가이드에 댓글이 달렸을 때 (회원만)
2. 운영자 공지 (최대 월 1회)
3. 좋아요 50개 도달 시 작성자에게 축하 알림 (회원만)

## 6. 상태 보존 — 글쓰기·검색은 “뒤로 갔다 와도 살아있어야 한다”

- 작성 중 콘텐츠는 **로컬 상태**(작성 폼)에 항상 살아있어야 한다.
- 모바일 패키징(Capacitor) 환경 기준 점검:
  - 글 작성 중 백그라운드 진입 → 복귀 시 본문이 유지되는가?
  - 뒤로 가기 → 다시 작성 페이지 진입 시 직전 입력이 복구되는가?
- 구현 방법: `GuideEditor`는 본문/제목/익명 정보를 `localStorage`에 자동 저장(드래프트), 페이지 마운트 시 복구한다.
  - 키: `guide-draft:new` (신규), `guide-draft:<id>` (수정)
  - 저장 성공 시 드래프트 비움
  - **모든 작성/입력 폼은 동일 패턴을 따른다.**

## 7. 사용자에게 보이는 문구는 “코드에 박지 않는다”

- 안내문, 공지, 약관, 문의 이메일은 모두 `site_settings` 테이블에 저장한다.
- 관리자 페이지(`/admin`)에서 즉시 수정 가능해야 한다.
- 코드에 들어가는 문구는 **레이아웃 라벨**(예: 버튼 텍스트) 정도에 한정한다.
- 새 페이지를 만들 때:
  - 본문이 “편집 가능해야 하는 안내”라면 site_settings 키를 추가하고 fallback 텍스트를 둔다.
  - 관리자 페이지의 `AdminSettingsForm`에 해당 키를 노출한다.

---

## 점검 체크리스트 (PR 머지 전)

- [ ] 모바일 360px 뷰포트에서 레이아웃 OK (원칙 1)
- [ ] 사용자 정보가 추가된다면 정책/삭제 페이지 본문도 갱신 (원칙 2)
- [ ] 새 알림이 있다면 시나리오 문서 등록 (원칙 5)
- [ ] 새 입력 폼은 드래프트 보존 패턴 적용 (원칙 6)
- [ ] 새 안내 문구는 site_settings 키로 추출 (원칙 7)
